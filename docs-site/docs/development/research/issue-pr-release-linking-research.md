# Issue-PR Release Linking Research

**Date:** 2026-03-13
**Status:** Implemented in v2.7.11 (PR #2317)
**Scope:** Enhance release notes generation to surface the originating GitHub issue alongside the PR that resolved it.

## Context

The project uses `.changelog/pr-NNN.txt` files — one per merged PR, generated by the Gemini-powered changelog generator in a GitHub Action — as the input for `generateReleaseNotes.mjs`. At the time this research began, only the PR number appeared in the final release notes. When a PR resolves a user-reported issue (e.g., PR #2310 closes issue #2293), the release notes gave no indication of the issue, making it harder for users to cross-reference bug reports with shipped fixes.

The goal was to enrich the `.changelog/` file format and the release notes generation pipeline to include `closes:` metadata, then surface it as clickable issue links in the release output.

## Finding: GraphQL Is Required

GitHub's REST API does not expose which issues a PR closes. The `closingIssuesReferences` field is only available through GitHub's GraphQL API. This is not a gap that can be worked around with REST — it is a deliberate API design difference.

The practical consequence is that the changelog-generator workflow must call the GraphQL endpoint to retrieve closing-issue data at the time the `.changelog/pr-NNN.txt` file is written. Fortunately, `github.graphql()` in `actions/github-script` works with the standard `GITHUB_TOKEN` that GitHub Actions already injects automatically. No new secrets, no additional permissions configuration, and no external API keys are needed.

The relevant GraphQL query shape is:

```graphql
query($owner: String!, $repo: String!, $pr: Int!) {
  repository(owner: $owner, name: $repo) {
    pullRequest(number: $pr) {
      closingIssuesReferences(first: 10) {
        nodes {
          number
          url
          title
        }
      }
    }
  }
}
```

## Proposed File Format

The updated `.changelog/pr-NNN.txt` format is backward compatible. Existing single-line files continue to work unchanged — the parser simply finds no `closes:` line and produces the same output as before.

New files written by the updated Action include a second line with closing issue metadata:

```
fix: only reload on main frame network errors, not sub-frames - by @IsmaelMartinez (#2310)
closes: #2293 https://github.com/IsmaelMartinez/teams-for-linux/issues/2293 Reload triggered on non-main-frame errors
```

If a PR closes multiple issues, each one appears on a separate `closes:` line.

## Files Changed

Four files are involved in this implementation:

`.github/workflows/changelog-generator.yml` is the entry point. The "Post-process summary" step is extended with a GraphQL query via `github-script`. The file-write is moved from a bash `echo` command into the JavaScript block to handle multi-line content safely. The file content is passed through an environment variable rather than directly interpolated into the YAML string, avoiding YAML injection if a commit message or issue title contains special characters.

`scripts/generateReleaseNotes.mjs` is updated so that `loadFromChangelogDir` returns objects of shape `{content, closingIssues}` rather than plain strings. The `closingIssues` array contains `{number, url}` entries parsed from the `closes:` lines. The `formatCategorizedEntries` function appends these as parenthesised Markdown links when present, producing entries like:

```
- fix: only reload on main frame network errors - by @IsmaelMartinez (#2310) ([#2293](https://github.com/IsmaelMartinez/teams-for-linux/issues/2293))
```

`scripts/release-prepare.mjs` is updated so that `loadChangelogEntries` takes only the first line of each `.changelog/` file. This ensures the `closes:` metadata lines are never included in `appdata.xml`, which should contain only human-readable changelog text for Linux software centres.

`.changelog/README.md` is updated to document the new two-line format, explain backward compatibility, and clarify that the `closes:` line is inserted automatically by the Action and should not be written manually.

## Backward Compatibility

The design is deliberately additive. Files written before this change contain only a single line and continue to parse correctly — the absence of a `closes:` line is treated as zero closing issues. Files written after the change contain the enriched format. No migration of historical `.changelog/` files is required.

## Output Example

Before this change, a release notes entry looks like:

```
- fix: only reload on main frame network errors, not sub-frames - by @IsmaelMartinez (#2310)
```

After:

```
- fix: only reload on main frame network errors, not sub-frames - by @IsmaelMartinez (#2310) ([#2293](https://github.com/IsmaelMartinez/teams-for-linux/issues/2293))
```
